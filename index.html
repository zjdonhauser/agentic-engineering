Concept 1 · Agentic Architecture

The Agentic Loop

Every agentic system — Claude Code, custom agents, the Agent SDK — runs the same fundamental cycle. The model reasons, optionally requests tools, you execute them, feed results back. stop_reason is the only reliable signal for loop control.

Send to Claude
→
Check stop_reason
→
tool_use?
→
Execute Tools
→
Append Results
↺
end_turn → Extract
claude-agent.ts:792 — runDiagnosticAgent()
for (let turn = 0; turn < MAX_TURNS; turn++) { const response = await client.messages.create({ model, max_tokens: 16384, system: systemPrompt, tools: TOOLS, messages, thinking: { type: "adaptive" }, output_config: { format: DIAGNOSTIC_OUTPUT_FORMAT }, }); messages.push({ role: "assistant", content: response.content }); if (response.stop_reason === "end_turn") { return extractResult(response.content); } // Execute all tool calls in parallel const toolUseBlocks = response.content.filter(b => b.type === "tool_use"); const toolResults = await Promise.all( toolUseBlocks.map(async (block) => ({ type: "tool_result", tool_use_id: block.id, content: await executeTool(block.name, block.input), })) ); messages.push({ role: "user", content: toolResults }); }
The signal
stop_reason drives everything

tool_use → execute, feed back, loop. end_turn → done, extract structured result. This is how Claude Code works internally too.

Parallel execution
Promise.all on tool calls

The model can request multiple tools in one turn. Execute them all in parallel — don't serialize what can be concurrent.

Anti-pattern
Never parse text to detect completion

Don't check for "DONE" in the response text. Don't set arbitrary iteration caps as the primary stop mechanism. stop_reason is the API contract.

Concept 2 · Context Management

Context Engineering

Agent output quality is bounded by input quality. Front-load cheap data before the first Claude call. DB queries cost milliseconds and zero tokens. Letting Claude discover context via tool calls costs seconds and thousands of tokens.

Concept 3 · Tool Design

Tool Descriptions Drive Selection

When the model picks the wrong tool, the correct fix is almost always: improve the description. Not few-shot examples (token overhead). Not a routing layer (over-engineered). Descriptions are the primary mechanism LLMs use for tool selection.

Bad — vague, overlapping
{ name: "get_sentry_event", description: "Get error data from Sentry" } { name: "search_sentry_issues", description: "Search for errors in Sentry" } // Both "get errors from Sentry" — which one? // Model can't tell when to use which.
Good — scoped, bounded
{ name: "get_sentry_event", description: "Fetch a specific Sentry error event by event ID. Returns exceptions with full stack traces, breadcrumbs, and HTTP request history." } { name: "search_sentry_issues", description: "Search Sentry for issues tagged with a specific job ID. Returns occurrence counts across ALL jobs sharing that fingerprint, not specific to this job." }
Principle
4-5 scoped tools, not 18 generic ones

Too many tools degrades selection reliability. Each tool has a single purpose and non-overlapping scope. Our agent has exactly 4.

Principle
Include boundaries and negative examples

"across ALL jobs, not specific to this job" — prevents misinterpretation. "Prefer get_github_file when you already know the path" — steers selection between similar tools.

Exam trap
Wrong tool selection? Fix the description.

The tempting answers: add few-shot examples (token overhead), build a routing layer (over-engineered). The correct answer on the exam and in practice: rewrite the description with clearer boundaries.

Prompt caching
cache_control on the last tool

{ type: "ephemeral" } on the last tool + system prompt creates a cache breakpoint. Sequential runs reuse the cached prefix (~5 min TTL).

Concept 4 · Prompt Engineering

Programmatic Enforcement

When must something happen? Don't ask the model. Prompt instructions are probabilistic — the model usually follows them. Schema enforcement is deterministic — the model can't deviate.

Bad — prompt-based (soft)
const response = await client.messages.create({ messages: [{ role: "user", content: "Diagnose this failure. Return JSON with summary and suggestedActions." }], }); // Returns: "Here's my analysis:\n```json..." // Or just prose. Non-deterministic.
Good — schema-enforced (hard)
const response = await client.messages.create({ messages: [{ role: "user", content: prompt }], output_config: { format: { type: "json_schema", schema: { /* your schema */ }, }}, }); // API guarantees valid JSON matching schema. // No parsing. Deterministic. Every time.
Our diagnostic schema
Enum-constrained actions + sourced evidence

type enum: retry | link_issue | create_issue | investigate | benign. source enum: sentry | job_context | datadog | tribal_knowledge. SHD consumes these programmatically.

Tighter for cheaper models
Haiku triage: decision: "update" | "escalate"

That's the entire output space. Can't hallucinate an action type. This tightness is what lets it run unsupervised.

Structured tool errors too
{ errorCategory, isRetryable }

Generic "tool failed" prevents the model from making recovery decisions. Structured errors with transient | not_found | validation | permission let it decide: retry, skip, or work with what it has.

const TRIAGE_OUTPUT_FORMAT = { type: "json_schema", schema: { properties: { decision: { enum: ["update", "escalate"], }, summary: { type: "string" }, rootCause: { type: "string" }, }, additionalProperties: false, }, };

Concept 6 · Reliability

The Autonomy ↔ Constraint Tradeoff

You earn autonomy by proving constraints are tight enough. The more programmatic gates around the output, the less human oversight you need. This applies to both API agents and Claude Code workflows.

Human-in-the-loop
Fully Autonomous
Interactive
Claude Code Sessions

You orchestrate, review every diff. CLAUDE.md provides conventions. The model proposes, you approve.

Gate: human approval per action

Semi Autonomous
SHD Morning Triage

Opus diagnoses overnight. Schema-enforced output. Dashboard shows evidence + actions. Human reviews and approves.

Gate: schema + human review

Autonomous
Haiku Repeat Handler

Error fingerprinted, matched to group. Haiku updates or escalates. No human until escalation threshold.

Gate: 2-value enum + sampling + daily caps

→ The concepts build on each other. Structured output + enum constraints + programmatic gates = the safety stack. And when attempts fail, prior findings are injected into retries (CCAF Task 1.7) so cheaper models build on earlier work instead of starting from scratch.
Concept 7 · Claude Code Workflows

Rules, Commands, and Team Patterns

Beyond CLAUDE.md — .claude/rules/ loads context conditionally by file path, and .claude/commands/ gives the whole team reusable workflows via git. Both are context engineering applied to the CLI.

.claude/rules/sequelize.md
--- paths: ["app/models/**", "app/migrations/**"] --- - New models use decorator pattern (@Table, @Column) - Register in typescriptModelsForSequelizeInitialization - Migrations always JS, always with down() - Wrap multi-step migrations in transaction - Don't use "as" aliases in includes - Prefer explicit property assignment over .update()
.claude/rules/fms-integrations.md
--- paths: ["lib/integrations/**"] --- - All integrations inherit from abstract.js - Check cubby/storedge/sitelink for existing patterns - Try/catch per unit, not whole batch - Fail loudly — throw, don't default to $0 - Use abstract methods before creating new ones
.claude/rules/testing.md
--- paths: ["**/*.test.ts", "**/*.test.tsx"] --- - Mocha + Chai + Sinon (not Jest) - sinon.restore() in afterEach - Stub at module boundary, not internals - Test behavior, not implementation
Why rules over one big CLAUDE.md
Glob frontmatter = scoped context

paths: ["lib/integrations/**"] means FMS patterns only load when you're editing integrations. Sequelize rules only load for models/migrations. Testing rules only for test files. Less noise, more relevant context per session.

.claude/commands/review-pr.md
# Review this PR against team patterns Check for: - No `any`, no `ts-ignore`, no `ts-expect-error` - Business logic in services, not controllers - Fail loudly for critical failures - Check existing patterns before adding new code - Validate parent resources before creating children Output scored issues with file:line references. // Committed to git → whole team gets /review-pr
Commands vs Skills
Simple markdown → full workflow

Commands (.claude/commands/): markdown prompts, shared via git, invoked with /command-name. Skills (.claude/skills/): SKILL.md with frontmatter — context: fork for isolation, allowed-tools for restrictions. Use skills when you need guardrails.

Takeaways

The Mental Models

01  The Agentic Loop

stop_reason drives everything. tool_use → execute → feed back. end_turn → done. This is the contract.

02  Context Engineering

Front-load cheap data. CLAUDE.md for CLI, buildJobContext() for APIs. Quality in = quality out.

03  Tool Descriptions

The #1 lever for tool selection. Include boundaries, negative examples, expected formats. 4-5 scoped tools > 18 generic ones.

04  Programmatic Enforcement

Schema > prompts for guarantees. output_config.format for structure. Enum constraints for cheaper models.

05  Cost ↔ Complexity

Opus for novel investigation. Haiku for known patterns. Prompt caching for repeated prefixes. Daily caps as guardrails.

06  Autonomy ↔ Constraint

More autonomy requires tighter programmatic gates. Earn unsupervised operation by proving the output space is constrained.
